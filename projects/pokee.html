Case Study · No. 01 / 03

科大讯飞
Pokee · 会议录音助手 Agent

这是一个从 0 到 1 的项目——为 AI 录音笔 Pokee 建立模型在会议录音场景下的意图理解与 Function Call 工具调用能力。我在其中独立输出规则框架、主导跨方对齐、管理 15 人标注团队，把"多轮开麦失效"这一典型 Badcase 从 AI 输出、训练数据、评测、业务四层彻底闭环。

01

背景  — the brief

What was the problem?

"科大讯飞 AI 录音笔 Pokee"是一台需要在会议场景里真正"懂你"的硬件——用户按下开麦键之后，它要识别对话意图、调用正确的工具、串联多个步骤，在复杂多轮交互里给出人性化回复。

项目启动时，模型在该场景下的工具调用能力几乎是空白。作为一个从 0 开始的训练数据建设任务，它需要同时回答三个问题：数据怎么建？规则怎么定？质量怎么保？

我的任务是和产品、工程、算法三方对齐，从多维度构建 Intent 与 Function Call 的训练数据，持续优化助手在复杂多轮与多工具串联场景下的调用准确性与回复质量。

02

我的动作  — what I did

Four parallel tracks

我把工作拆成四条并行的线索，每一条都需要独立推进，又必须最终汇合到同一个闭环里。

  1. 规则框架从 0 开始写。独立输出初版规则文档，针对 Intent 理解与 Function Call 调用两大类场景制定标注定义、边界条件和反例集合。
  2. 爬坡期的跨团队联调。每周 3 次跨团队联调会，把产品的交付目标、工程的接口约束、算法的能力评估三方的语言统一到同一份规则里——推动规则对齐与问题收敛。
  3. 标注团队与闭环。独立管理 15 人标注团队，设计 3-5 天准入考核机制 与 9 步标注闭环，人均日产能稳定在 10-15 条，保障按期交付。
  4. Badcase 四层归因。针对"多轮开麦失效"这一典型问题，从规则、数据、执行、工程四个层面同时排查，最终定位到"规则盲区 + 工程判断与规则不对齐"两个并行原因。
真正的闭环不是"改了规则就完事"——它要从 AI 输出、训练数据、模型评测、线上业务四层都验证一遍。
03

评测支撑  — evaluation

How did we measure?

光有数据不够，还要知道模型"是不是真的变好了"。我针对录音场景的关键能力点——多工具串联、多轮开麦、语言区分、异常场景——构建专项评测集，在 Function Call 调用准确率、Intent 理解准确率、多轮上下文一致性 等核心维度下，量化评估模型版本迭代效果差异。

评测结果会反哺到规则和数据侧，形成"规则 → 数据 → 训练 → 评测 → 规则"的完整闭环。

这套机制让我们在 Badcase 修复之后，能在 AI 输出层、训练数据层、评测层、下游业务层分别验证——确保问题真正被修掉，而不是"看起来被修掉"。
